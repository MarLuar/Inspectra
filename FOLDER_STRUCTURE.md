# InSpectra Project Folder Structure

This document explains the purpose of each folder in the InSpectra project structure.

## Root Level Folders

### `.dart_tool/`
- Contains Dart build system cache and temporary files
- Generated by the Dart toolchain during development
- Should be ignored by version control (already in .gitignore)

### `.git/`
- Git repository metadata and history
- Contains all version control information
- Automatically created when initializing a Git repository

### `.idea/`
- IDE-specific configuration files for IntelliJ IDEA/WebStorm
- Contains editor settings, code styles, and project preferences
- Should typically be ignored by version control

### `android/`
- Contains the native Android project files
- Includes Gradle build scripts, manifest, and resource files
- Allows Flutter to build and deploy to Android devices
- Contains platform-specific code and configurations

### `assets/`
- Stores static assets used by the application
- Contains images, fonts, and other media files
- Referenced in the pubspec.yaml file for inclusion in the app
- Organized into subfolders like `images/` for better organization

### `build/`
- Contains compiled application files and build artifacts
- Generated during the build process
- Should be ignored by version control (already in .gitignore)
- Includes intermediate build files for various platforms and dependencies
- Contains platform-specific build outputs (APKs, AABs, etc.)
- May include dependency-specific build folders like `image_picker_android/`
- Contains subdirectories for:
  - `image_picker_android/`: Build artifacts for the image picker plugin
    - `intermediates/`: Temporary files generated during the build process
      - `manifest_merge_blame_file/`: Manifest merging information for debugging
      - `aar_main_jar/`: Compiled JAR files for the Android Archive
      - `merged_manifest/`: Combined Android manifest files
      - `compile_r_class_jar/`: Compiled R.java class files
      - `compile_library_classes_jar/`: Compiled library class files
      - `library_assets/`: Assets from the library

### `ios/`
- Contains the native iOS project files
- Allows Flutter to build and deploy to iOS devices
- Includes Xcode project files, Info.plist, and resource files
- Contains platform-specific code and configurations

### `lib/`
- Contains the main Flutter/Dart source code
- This is where the core application logic resides
- Organized into subfolders for better code organization
- The entry point is `main.dart` in this directory

### `linux/`
- Contains the native Linux project files
- Allows Flutter to build and deploy to Linux desktop
- Includes CMake build files and platform-specific code

### `macos/`
- Contains the native macOS project files
- Allows Flutter to build and deploy to macOS desktop
- Includes Xcode project files and platform-specific code

### `test/`
- Contains unit, widget, and integration tests
- Used for automated testing of the application
- Helps ensure code quality and prevent regressions
- Follows Flutter's testing conventions

### `web/`
- Contains web-specific configuration and assets
- Allows Flutter to build and deploy to web browsers
- Includes index.html and other web-specific files

### `windows/`
- Contains the native Windows project files
- Allows Flutter to build and deploy to Windows desktop
- Includes MSBuild files and platform-specific code

## Source Code Subfolders (`lib/`)

### `config/`
- Contains configuration files for external services
- Includes Firebase configuration and other setup files
- Separates configuration from business logic
- Contains template files for sensitive configurations

### `models/`
- Defines data models and business objects
- Contains classes like Project and Document
- Represents the data structure of the application
- Includes serialization methods (toJson/fromJson)

### `providers/`
- Contains state management providers (currently empty)
- Would typically hold ChangeNotifier classes for Provider pattern
- Manages application state and data flow
- Enables reactive UI updates

### `screens/`
- Contains UI screens and pages of the application
- Each screen represents a complete user interface
- Includes navigation and user interaction logic
- Examples: home_screen.dart, project_detail_screen.dart

### `services/`
- Contains business logic and service classes
- Handles operations like document scanning, database access, and cloud sync
- Encapsulates complex functionality in reusable modules
- Examples: database_service.dart, document_scanner_service.dart

### `utils/`
- Contains utility functions and helper classes
- Reusable code that doesn't belong to specific features
- Common functions for formatting, validation, etc.
- (Currently empty in this project)

### `widgets/`
- Contains reusable UI components
- Custom Flutter widgets used across multiple screens
- Promotes code reuse and consistent UI
- (Currently empty in this project)

## Root Level Files

### `pubspec.yaml`
- Flutter project configuration file
- Defines dependencies, assets, and project metadata
- Controls how assets are included in the application
- Specifies the Flutter SDK version and other settings

### `pubspec.lock`
- Lock file for pub dependencies
- Ensures consistent dependency versions across environments
- Automatically generated and updated by pub

### `README.md`
- Project documentation and getting started guide
- Explains the purpose and features of the application
- Provides instructions for building and running the app

### `.gitignore`
- Specifies files and folders to be ignored by Git
- Prevents temporary files and build artifacts from being committed
- Keeps the repository clean and focused on source code

### `.metadata`
- Internal Flutter tooling file
- Tracks project-specific metadata
- Should be ignored by version control

### `analysis_options.yaml`
- Static analysis configuration for Dart code
- Defines linting rules and code quality standards
- Helps maintain consistent code style across the project

### `COMPLEXITY_AND_FEATURES.md`
- Documentation about complex features and implementation details
- Explains challenging aspects of the application
- Provides insights into advanced functionality

### `FEATURE_ENHANCEMENTS.md`
- Documentation about planned feature enhancements
- Tracks ideas for future improvements
- Helps prioritize development efforts

### `main.dart`
- The entry point of the Flutter application
- Contains the main() function and InSpectraApp widget
- Initializes services and sets up the app's initial route
